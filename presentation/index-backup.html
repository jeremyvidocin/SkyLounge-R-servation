03 - Deep Dive

Anti-Double Booking

Le problème critique de tout système de réservation

⚠

Le Problème

👤 User A
14:00:00 - Sélectionne le 15 janvier
14:00:02 - Remplit le formulaire
14:00:30 - Paie
👤 User B
14:00:01 - Sélectionne le 15 janvier
14:00:03 - Remplit le formulaire
14:00:25 - Paie
💥 Double réservation !
🔒

Ma Solution : Locks avec TTL

// WordPress Transient = Lock temporaire
set_transient('cw_locks_' . $offre_id, [
    'token'   => 'uuid-unique',
    'start'   => '2025-01-15',
    'end'     => '2025-01-15',
    'expires' => time() + 1200  // 20 min
], 1200);

// TTL adaptatif selon capacité
$ttl = ($capacity == 1) ? 1200 : 300;
// 20 min bureau privé, 5 min salle partagée

// Anonymisation IP - RGPD compliant
function cw_anonymize_ip($ip) {
    if (filter_var($ip, FILTER_VALIDATE_IP, FILTER_FLAG_IPV4)) {
        return preg_replace('/\.\d+$/', '.0', $ip);
        // 192.168.1.123 → 192.168.1.0
    }
    return preg_replace('/:[^:]+$/', ':0000', $ip);
}

04 - Difficultés

Problèmes Rencontrés

💥

Double Booking Race Condition

Deux utilisateurs pouvaient réserver le même créneau simultanément

💡
Solution : Système de locks avec transients WordPress et TTL adaptatif
💥

Cohérence JSON / CPT

Le cache JSON pouvait se désynchroniser des vraies réservations

💡
Solution : CRON quotidien de vérification + rebuild automatique
💥

Guest Checkout WooCommerce

Simplifier le tunnel sans perdre les données de réservation

💡
Solution : Metadata sur order items + hooks personnalisés
💥

Génération PDF sans plugin payant

Créer des contrats pro sans DOMPDF/mPDF

💡
Solution : Template HTML + fallback WPO WCPDF si disponible
